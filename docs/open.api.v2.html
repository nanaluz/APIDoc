Integration guide

Access URLs

Please make API requests with the following base urls:

  • https://www.mxc.ceo
  • https://www.mxc.co
  • https://www.mxcio.co

Request Format

The APIs accept GET, POST or DELETE request, as will be described in later sections

  • For a GET or DELETE reqeust, all parameters are path paremeters (query string)
  • For a POST reqeust, business parameters are to be put in request body and of JSON format, they will not be used for signing of the request. To calculate signature, params api_key、req_time、recv_window(optional)、sign need to be sent as path parameter

Response Format

Response is of JSON format for all APIs

Signature

Public APIs (for market data) and private APIs (for account and order operations) are provided.

Signature is not required for public APIs, but access key from API key is required for every API request.

And for private APIs, signature is required and will be used for checking the validity of each request.

Please refer to Signature Method for detail

Time Security

 if (req_time > (server_time + 2 ) || req_time < (server_time - recv_window)) {
   // reject request     
 }else{
  // process request
 }

Parameter req_time is required for each private API, it's a string of 10 digits stands for the seconds since Unix Epoch. After API server receives a request, timestamp validation will be conducted to make sure the requesting time from client is valid. If req_time is behind the sever time for more than 5 seconds (the default value, this value can be altered by sending an optional parameter recv_window, the max allowed value of this parameter is 60 seconds, it's not recommended to use values greater than 30 seconds) or is ahead of the server time for more than 2 seconds, the request is considered invalid

See the pseudocode to get the clear idea

Signature Parameters

Parameters related to request signature

Components Description
param api_key access key from API key
param req_time the timestamp when sending the request, string of 10 digits stands for the seconds since Unix epoch, for instance 1572537600
param sign calculated signature
HTTP request method GET, POST or DELETE as described in later sections for each API
URI URI of the API
business parameters business parameters required by each API, only the business parameters of GET and DELETE requests will be included in the signature calulation

Creating API key

API key can be created in the User Center of MXC site. There are two parts of API key, Access key and Secret key，both of them will be used for calculating and validating a signature

Secret key will ONLY be displayed in the page when API key is generated, please make records accordingly

Signature Method

Signing target of a GET or DELETE request 

GET
/api/v2/order/open_orders
api_key=mxcV9JCC8iu8zpaiWC&limit=1000&req_time=1572936251&startTime=1572076703000&symbol=MX_ETH&tradeType=BID

The actual request after signed 

GET https://www.mxc.co/api/v2/order/open_orders?api_key=mxcV9JCC8iu8zpaiWC&limit=1000&req_time=1572936251&startTime=1572076703000&symbol=MX_ETH&tradeType=BID&sign=0d1a5ee9e76a4907abedc4fe383ddfd0

Signing target of a POST request

POST
/api/v2/order/place_batch
api_key=mxcV9JCC8iu8zpaiWC&req_time=1572936038

First we need to compose the target string, each part will be separated by line break \n

1. HTTP request method, GET, DELETE or POST
2. Request URI
3. Order the request parameters by name (asc): api_key, req_time, recv_window（optional）and business parameters. Concatenate the parameters with &, at last append the secret key at the end of the string. As a result the string will be used for signing (when there is special characters like comma in parameter value, as in the batch processing APIs, these special characters need to be URL encoded)

For GET or DEELTE requests, all reqeust parameters will be included in the signature calculation. For POST requests, only HTTP request type, URI, api_key, req_time and the optional recv_window will be included for signature calculation

HMAC SHA256 signature is required for the private APIs, by doing the hmac sha256 calculation against the string mentioned above.

The signature will be put as the value of parameter sign, and be included in the request

Error Code

The following error information can be returend

Code Description
400 Invalid parameter
401 Invalid signature, fail to pass the validation
429 Too many requests, rate limit rule is violated
10072 Invalid access key
10073 Invalid request time
30000 Trading is suspended for the requested symbol
30001 Current trading type (bid or ask) is not allowed
30002 Invalid trading amount, smaller than the symbol minimum trading amount
30003 Invalid trading amount, greater than the symbol maximum trading amount
30004 Insufficient balance
30005 Oversell error
30010 Price out of allowed range
30016 Market is closed
30019 Orders count over limit for batch processing
30020 Restricted symbol, API access is not allowed for the time being

Rate Limit

There is rate limit for API access frequency, upon exceed client will get code 429: Too many requests.

Access key from API key is required for every API access, the access key is used as the basic unit of rate limit.

Different API comes with different settings.

The rate limit detail of each interface can be get from Rate Limit Info

  • POST /open/api/v2/order/place_batch

Request parameters：

Parameter Data type Mandatory Description Allowed range
symbol string Y symbol name
price string Y price
quantity string Y quantity of the order
trade_type string Y trade type BID，ASK
order_type string Y order type LIMIT_ORDER，POST_ONLY，IMMEDIATE_OR_CANCEL
client_order_id string N client order id, can be used to differentiate the orders in a batch(for example can be an UUID)

Response：

Field Data type Description
data map client order id and the corresponding new order id
For each batch request, the parameter client_order_id should be consistent, meaning if any of the order contains client_order_id, then all orders should contain client_order_id as well. And please be noted the client_order_id will not be stored, it will only be used for checking the relationship between orders in request and newly created order id
